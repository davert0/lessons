1. Модуль получения данных от поставщиков услуг.
Контекст:
    Задача модуля - после запроса на получение данных в нашем приложении - асинхронно получить данные у поставщика услуг, сходив по нескольким ручкам, и собрав из них целостный доменный объект. Особенность 
    апи заключалось в том, что было несколько ручек, которые отдавали список объектов (счета, балансы счетов, карты, балансы карт) независимо друг от друга. И вот нужно было взять 4 списка и смержить их в один
    валидный список счетов

До:
    В целом, у данного модуля была довольно неплохая cohesion, но при этом внутри самого модуля была крайне трудно вносить какие-либо изменения, поток данных был непредсказуем,
    приватные методы внутри себя содержали довольно ветвистые if else деревья для разных случаев внутри одного и того же модуля.
После:
    Я выделил несколько логических юнитов - 
    1. Непосредственно адаптеры для получения данных по каждой из ручек
    2. Общий модуль tokenManager для управления парой AT\RT для предотвращения состояний гонок (refresh token ротируется на каждое обновление AT).
    3. Builder - собирающий валидный объект
    4. Repo - сохранаяющий валидный объект
    5. Непосредственно runner, собирающий первые четыре юнита в одно целое

Написал интеграционные тесты, покрывающие все комбинации, когда объект, собранный из ответа разных ручек считается валидным и сохраняется в БД.


2. API Composition сервиса сделок
Контекст:
    Нужно отдать обогащенный ответ от сервиса сделок на фронт, включающий в себя данные пользователя, гео, мету объявления.

До:
    Данные собирались внутри сервиса сделок, что создавало перекрестные зависимости между сервисами, логически не связанными друг с другом
После:
    1. Выделил отдельный API Composition слой, собирающий данные из нескольких сервисов для фронта
    2. Фронт завязался на Openapi контракт, с возможностью кодогенерации клиентского кода
    3. Исчез ряд зависимостей между доменными сервисами


В тестах появилась возможность избавиться от моков сторонних сервисов и сконцентрироваться на доменной логике сервиса сделок

3. Отдельный сервис для домена агентов и агенств недвижимости
Контекст:
    В рамках сервиса сделок сформировался поддомен агентов и агенств. Вначале развития продукта функционал был достаточно простым - 
    отдельная роль пользователя и проверка прав доступа основанная на роли. В дальнейшем функционал начал разрастаться, появились агенства,
    с возможностью для руководителя добавлять/удалять сотрудников, назначать на них сделки, распределять по скиллам и тд. Далее потребность
    в таком же функционале возникла и у соседних команд, фичареквесты становились дороже, в связи с чем было принято решение выделить этот домен 
    в отдельный сервис.

До:
    Отдельный крупный модуль со своей бизнес-логикой внутри сервиса сделок
Рефакторинг:
    1. Выделил основные пользовательские сценарии на момент принятия решении о переносе.
    2. Создал ограниченный контекст - вынес все связанное с агенствами в отдельный модуль.
    3. Перевел взаимодействие всех остальных частей сервиса сделок только через API контракт модуля агенств.
    4. Создал сервис, продублировал в нем функционал с существующего модуля.
    5. Написал адаптеры в оригинальном сервисе.
    6. Бесшовно переключил использование внутреннего модуля на внешний.
После:
    Был создан отдельный сервис, с чистой доменной областью и возможностью для расширения. 

В ходе выделения ограниченного контекста были написаны интеграционные тесты на основные сценарии основного сервиса сделок, в которых 
участвовала работа с агенствами. Одним из критериев успеха было сохранение работоспобности всех интеграционников после переноса.
